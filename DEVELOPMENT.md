# 开发任务计划书 (Development Roadmap)

## 1. 核心开发原则
- **Demo-First**: 每一个功能模块必须先实现原型Demo，经验收确认后，再进行正式的功能工程化实现。
- **数据驱动**: 严格遵守 `产品编号` 作为唯一键的关联逻辑。
- **质量闭环**: 任务执行 $\rightarrow$ Checklist核对 $\rightarrow$ 执行日志记录 $\rightarrow$ 验收确认。

## 2. 分阶段任务分解与验证列表

### 阶段一：基础设施与物料主库 (Foundation & Master DB)
**目标**: 建立Tauri+SQLite环境，实现采购计划到主库的同步逻辑。
- [ ] **T1.1 环境搭建**: 初始化Tauri v2项目，配置SQLite3本地数据库。
- [ ] **T1.2 数据库建模**: 设计`Purchase` (采购表) 与 `MaterialMaster` (产品信息表) 结构。
- [ ] **T1.3 Demo-1 (同步验证)**: 
    - 实现简单采购录入界面。
    - 验证：录入采购项 $\rightarrow$ 自动同步至主库 $\rightarrow$ 主库查询可见。
- [ ] **验收标准**: 
    - [ ] 数据库文件正常生成。
    - [ ] 采购表 $\rightarrow$ 主库的单向同步无延迟且数据准确。

### 阶段二：出入库流水记录 (Transaction Flow)
**目标**: 实现基于主库的快捷录入，确保“先定义后使用”。
- [ ] **T2.1 数据库建模**: 设计`Inbound` (入库表) 与 `Outbound` (出库表) 结构。
- [ ] **T2.2 Demo-2 (快捷录入验证)**:
    - 实现入/出库录入界面。
    - 验证：输入`产品编号` $\rightarrow$ 自动带出名称/规格 $\rightarrow$ 仅录入数量/日期。
- [ ] **T2.3 约束验证**: 验证输入不存在的`产品编号`时系统应拦截。
- [ ] **验收标准**: 
    - [ ] 录入效率提升（无需重复输入物料详情）。
    - [ ] 强制执行“先定义后使用”逻辑。

### 阶段三：库存实时汇总 (Inventory Summary)
**目标**: 实现全系统的最终输出，验证数据聚合正确性。
- [ ] **T3.1 聚合逻辑实现**: 编写SQL视图或函数，计算 $\sum \text{入库} - \sum \text{出库}$。
- [ ] **T3.2 Demo-3 (报表验证)**:
    - 实现库存汇总表界面。
    - 验证：修改入出库流水 $\rightarrow$ 汇总表期末库存实时更新。
- [ ] **验收标准**: 
    - [ ] 汇总数据与流水账单完全一致。
    - [ ] 响应速度满足本地实时查询要求。

### 阶段四：附件管理与系统完备 (Finalization)
**目标**: 完成本地附件存储及整体 UI/UX 优化。
- [ ] **T4.1 附件存储实现**: 实现物料附件在本地文件夹的映射与保存。
- [ ] **T4.2 完整CRUD**: 为所有模块增加编辑、删除、搜索功能。
- [ ] **T4.3 跨平台适配**: 优化 macOS/Windows 界面兼容性。
- [ ] **验收标准**: 
    - [ ] 附件可正常打开且路径不失效。
    - [ ] 全链路业务流程（采购 $\rightarrow$ 主库 $\rightarrow$ 入出 $\rightarrow$ 汇总）闭环流畅。

## 3. 任务执行模板
每次任务启动时，Agent 需在日志中包含：
1. **当前任务 ID**: (如 T1.3)
2. **执行 Checklist**: [ ] 步骤A $\rightarrow$ [ ] 步骤B
3. **Demo 验证点**: 预期结果 vs 实际结果
4. **执行日志**: 关键操作记录及遇到的问题
